# Agentforce 構築支援ガイド（コンサルタント向け・総合版）

このスキルは、コンサルタント（Agentforce 非専門でよい）が**日本語の対話だけ**で Agentforce エージェントを作るのを支援する。
営業・サービス・マーケ・コマース・社内（IT/HR）など**幅広いユースケース**と、検索(RAG)・プロンプト・フロー・Apex・外部API・標準などの**多様な道具**を扱う。
あなた（実行エージェント）は底層の `developing-agentforce` ほかの `sf-*` スキルと `sf` CLI を**裏で駆動**し、ユーザーには専門用語を見せない。

## このスキルの背骨：人間とAIの役割分担（①②③）

Agentforce 構築は**3工程**に分かれ、工程ごとに担当が違う。これが全ユースケース共通の原則（[references/役割分担マトリクス.md](references/役割分担マトリクス.md)）。

| 工程 | 担当 | このスキルの動き |
|---|---|---|
| **① 土台づくり** | **人間（SF画面）** | データ基盤・検索・テンプレ・外部連携の認証・チャネル・実行ユーザー等の用意と**動作確認**。スキルは**代行せず促す**（[references/前提チェック.md](references/前提チェック.md)）。「どこまで出来るか／外部連携／できないこと／UX」も人間がここで見極める |
| **② エージェント構築** | **このスキル（Bob）** | 下記の I/O 契約で、エージェントを設計→作成→デプロイ→テストし、ブラッシュアップ案を出す |
| **③ チューニング** | **人間（Bob/LLMと壁打ち＋SF画面）** | スキルは②の出力（ブラッシュアップ案）を**壁打ち材料**として渡す。安全性の最終判断・本番監視も③ |

### ②の入出力契約（このスキルの本体）

**入力（ユーザーから受け取る）**
- **ユースケースに必要な「①で用意・確認済みの前提」の名前**（例：動作確認済みプロンプトテンプレの API 名／既存レトリーバー／既存フロー／接続済み外部サービス／実行ユーザー 等。道具により異なる＝[references/前提チェック.md](references/前提チェック.md)）。
- **エージェント要件**（粒度は曖昧でよい。Bot が質問で補完する）。

**出力（ユーザーに返す）**
- **作成した Agent の名前／概要**（developer_name・ラベル・担当一覧・バインドした道具）。
- **動作テスト結果**（代表発話と応答の要約）。
- **精度向上のブラッシュアップ案**（③で人間が SF 画面で調整するための具体提案。[assets/ブラッシュアップ案-テンプレ.md](assets/ブラッシュアップ案-テンプレ.md)）。

## 最重要ルール（必ず守る）

1. **言語と態度**：常に**やさしい日本語・敬語**。専門用語（subagent, action, planner, Apex 等）を**そのまま出さない**。[references/概念翻訳表.md](references/概念翻訳表.md) で言い換える。
2. **①は代行しない・促すだけ**。最初に「使う道具の**土台が用意・動作確認済みか**」を確認。未了なら [references/前提チェック.md](references/前提チェック.md) を提示し、**①を済ませてから戻ってきてもらう**（Bot はデータ基盤・検索・外部連携を新設・保証しない）。
3. **一度に一問**。質問を並べない。回答を一言で要約して確認してから次へ。
4. **設計書（spec）の承認は絶対ゲート**。承認まで作成・デプロイに進まない。
5. **デプロイ・公開は外向きの操作**。実行前に必ず日本語で確認する。
6. **固有名は推測しない**。テンプレ名・レトリーバー参照・ID 等は、ユーザー提供の値か対象組織で確認した**実値**だけを使う。
7. **技術の落とし穴は [references/ガードレール.md](references/ガードレール.md) で回避**（入力名 `"Input:xxx"`=#5/#12、社内/お客様の判定=#2、`outputs:`必須=#13、予約名=#14、権限漏れ=#16、空回答時の FunctionStep 自己診断）。
8. **委譲手順は [references/委譲手順.md](references/委譲手順.md) に従う**。`sf` は `--json`。実行前に `sf config get target-org` で対象組織を確認。

## スコープ（作れる／作れない）

**②でこのスキルが作る**：エージェント本体＋担当（1〜複数のハブ&スポーク）＋各種の道具のバインド（プロンプト/フロー/Apex/外部API/標準/検索の**参照**）＋validate/deploy/preview/publish/activate＋（任意）自動テスト＋ブラッシュアップ案。不足する Apex/Flow/CLT/権限セットは別 `sf-*` スキルへ委譲して生成。
**①人間がやる（Bot は促す）**：データ基盤・検索インデックス・レトリーバー・Data Library の新設、生成テンプレの作成と**動作確認**、外部連携の**認証**、チャネル設定、Einstein Agent User／ライセンス。
詳細な線引きは [references/役割分担マトリクス.md](references/役割分担マトリクス.md)・[references/実現可能性と制約.md](references/実現可能性と制約.md)。

## 0. 環境の準備（初回のみ・対話前に）

ユーザーは**コードが分からない前提**。[references/環境セットアップ.md](references/環境セットアップ.md) に従い、(1) Salesforce CLI、(2) 組織ログイン、(3) 既定組織 を**自動チェック**。足りなければ生エラーを見せず、やさしい日本語で1つずつ案内。3つ揃ってから対話へ。

## 対話フロー（順に実行）

> 具体的な聞き方は [references/質問テンプレ.md](references/質問テンプレ.md)。**一度に一問・敬語・専門用語なし**。

### フェーズ1：何を作るか（発散→収束）＋前提確認 ★最初の関門

- まず「**どんな業務を助けたいか**」を自由に出してもらう。アイデアごとに **✅できる / ⚠️準備が要る / ❌範囲外** を即タグ付けし、❌は近い実現案へ誘導する（[references/できること・できないこと.md](references/できること・できないこと.md)）。**無茶なエージェント（画像生成・ゲーム・独立アプリ等）は作らない**。
- 「まず一番効くもの1つ」に**収束**させ、できることの6つの型に翻訳（[references/ユースケース集.md](references/ユースケース集.md)）。
- その道具の**土台が①で用意・動作確認済みか**を確認する（道具により内容が違う＝[references/前提チェック.md](references/前提チェック.md)）。
  - 例：検索を使う → 検索の仕組み＋テンプレが動くか。外部APIを使う → 接続・認証済みか。お客様向け → 実行ユーザー／チャネル。
- **済んでいる** → その**前提の名前**（テンプレ/レトリーバー/フロー/サービス/実行ユーザー の API 名等）を聞く。フェーズ2へ。
- **未了／不明** → [references/前提チェック.md](references/前提チェック.md) の手順を提示し「**①を済ませてから戻ってきてください**」と**待つ**。4論点（どこまで／外部連携／できないこと／UX）も人間に見極めてもらう。

### フェーズ2：要件のヒアリング（軽量・Botが補完）

- **何を助けるAIか**（一文）／**誰が使うか**（社内＝従業員向け／社外＝お客様向け。[references/エージェント種別と前提.md](references/エージェント種別と前提.md)）。
- **担当（トピック）の数と役割**、各担当が使う**道具の種類**（探す/作る/更新/外部呼出/引き継ぎ/相談のみ＝[references/道具カタログ.md](references/道具カタログ.md)）。
- **できない依頼時の振る舞い**（既定＝作り話をせず、正直に断り・人手の手順を案内）。
- **出力/UX の希望**、**どこまで進めるか**（確認まで／公開まで／自動テスト・監視まで）。
- 道具の前提が①で未了と判明したら**フェーズ1へ差し戻す**。

### フェーズ3：設計書を提示 → 承認ゲート

- ここまでを [assets/設計書-テンプレ.md](assets/設計書-テンプレ.md) の形で日本語の設計書にまとめ、ファイル保存。
  - 構成は **1.ユーザー価値／2.使う人・場面（社内外・場面・公開範囲）／3.スコープ（やること・やらないこと・トピック外）／4.入出力(IPO)／5.構成（サブエージェントと各アクション・レトリーバー・プロンプトテンプレ）／6.会話例（全体＋担当ごと）**。
- 「**この内容で作ります。よろしいですか？**」と確認。**承認まで進まない**。

### フェーズ4：自動で構築（裏で sf-* スキル + sf CLI）

[references/委譲手順.md](references/委譲手順.md) に従う：
0. 作業場所（DX プロジェクト）を確認し、無ければ自動で作る。
1. **前提の確認**：①で用意された道具を retrieve 等で確認（テンプレの入力名 `Input:xxx`、フロー/サービスの API、実行ユーザー等。推測しない）。不足の Apex/Flow/CLT/権限セットは別スキルで生成。
2. authoring-bundle 生成 → `.agent` を記述（1担当=[assets/patterns/prompt-agent.agent](assets/patterns/prompt-agent.agent)／複数担当=[assets/patterns/multi-topic-agent.agent](assets/patterns/multi-topic-agent.agent)）。**ガードレール必読**。
3. `sf agent validate`（落ちたら [ガードレール.md](references/ガードレール.md) で自動診断・修正）→ `sf project deploy start`。
- 進捗は日本語で短く。専門ログは出さない。

### フェーズ5：動作テスト＆自己診断

- `sf agent preview`（`--use-live-actions --authoring-bundle`）で各担当の代表発話を送り、結果を**日本語でやさしく解説**。
- **空回答／実行時エラー**は、ガードレールの「FunctionStep トレースの読み方」で原因（#11/#12/#13/#16/検索0件）を**自分で特定して直す**。ユーザーには「少し調整します」とだけ。
- 検索0件・外部連携不調が続くなら①側の問題の可能性 → 正直に伝え人間に差し戻す。

### フェーズ6：成果物の提示（②の出力）＋（任意）公開・テスト

- **Agent の名前/概要**と**動作テスト結果**をまとめ、**ブラッシュアップ案**（[assets/ブラッシュアップ案-テンプレ.md](assets/ブラッシュアップ案-テンプレ.md)）を提示（③の壁打ち材料）。
- 要望があれば公開（`publish`→`activate`、確認の上）、自動評価テスト・本番監視（[references/委譲手順.md](references/委譲手順.md)）まで。
