# 前提チェック（①②の土台は人・AIは促すだけ）

組み立て（④）に入る前に、**人が①②（土台づくり）を済ませている**必要がある。AI は土台を代行しない・検索精度や外部連携を保証しない。
入口A/Bとも、以下を**やさしい日本語**で確認し、未了なら手順を渡して**待つ**。

## ブレストでの境界タグ（入口B用・✅/⚠️/❌）

アイデアを即タグ付けして誘導する（「できる/できない」はAIが既知。重く文書化しない）。
- **✅ できる**（会話で完結）：調べて答える／文章を作る／記録を作る・更新／決まった処理／連携済み外部呼出／人へ引き継ぎ。「それはできます」。
- **⚠️ 準備が要る（グレー）**：下表のとおり①②の土台が要る。「できます。ただ先に〇〇の準備が要ります」。
- **❌ 範囲外**：画像/動画/音の生成・自由なアプリUI/ゲーム・独立アプリ・リアルタイム制御・厳密計算の最終確定 等。「それは範囲外ですが、近いところで〇〇なら」。

## 前提は「使う道具」によって変わる（⚠️の中身）

| 使う道具 | ①②で人が用意・確認するもの | ②で受け取る入力 |
|---|---|---|
| 文章を作る（プロンプト道具） | テンプレ作成＋**Prompt Builder で動作確認**＋有効化 | テンプレの **API 名** |
| 検索（レトリーバー/RAG） | **Data Cloud・検索インデックス・レトリーバー**作成＋確認 | レトリーバー名（or それを使う検証済みテンプレ名） |
| ナレッジ/Data Library | ナレッジ取り込み・**Data Library** 作成・紐付け | 紐付け済みであること |
| 外部API | **Named Credential の認証**（OAuth/シークレット）・接続承認 | サービス名／接続済みであること |
| 人にバトンタッチ | **Omni-Channel/メッセージング**設定（お客様向け） | チャネル設定済みであること |
| お客様向けエージェント | **Einstein Agent User 作成＋ライセンス**（詳細は公式 `developing-agentforce`） | 実行ユーザー名 |
| フロー/Apex 道具 | （任意）画面で作る場合のみ | 既存なら API 名。無ければ④で生成可 |

> 下は**最も多い「検索を使う生成テンプレ」**の確認手順。他の道具も考え方は同じ（①②で用意・確認 → ④へ名前を渡す）。

## なぜ①②は人がやるのか

- **レトリーバー（検索インデックス）は CLI で一覧も新設も（このスキルでは）できない**。既存の検索の仕組みを使うのが前提。
- 「**どこまで出来るか／外部データ／できないこと／UX**」の4論点は、**人が Prompt Builder（SF画面）でテンプレを実行して、検索が効くか・どんな結果が返るかを目で見て判断**するのが最も確実。AI の preview では検索データの良し悪しまでは保証できない。

## ユーザーに渡すチェックリスト（①の完了条件）

「お手数ですが、エージェントを作る前に、次がSF画面で**確認済み**かを教えてください」：

1. **レトリーバー（検索の仕組み）が設定済み**である（過去データ・資料の検索元）。
2. その検索を使う**プロンプトテンプレートが作成済み**で、**Prompt Builder の「プレビュー/テスト」で実際に検索が効き、期待する結果が返る**ことを確認した。
   - = テンプレ本文に検索結果（`{!$EinsteinSearch:….results}`）と入力（`{!$Input:…}`）が入っていて、テストで中身のある回答が出る状態。
3. テンプレが **有効（Active）** になっている（プレビューで動く＝有効なはず）。
4. 4論点を**見極めた**：
   - **どこまで出来るか**：このテンプレで返る範囲（何が答えられて何が答えられないか）。
   - **Box 等の外部データ**：使うなら、それは**すでに検索に取り込まれている**か（新規連携は範囲外）。
   - **できないこと**：テンプレが答えられない依頼の傾向。
   - **UX**：出力の言語・形式・長さの希望。
5. その**プロンプトテンプレートの API 名**が分かる（Bot への入力になる）。
   - SF: セットアップ →「プロンプトテンプレート」→ 対象テンプレ → API 参照名。

## 受け答えの分岐

- **すべて確認済み＋API名あり** → 「ありがとうございます。その『<API名>』を使ってエージェントを作ります」とフェーズ2へ。
- **テンプレはあるが動作未確認** → 「先に Prompt Builder のプレビューで**検索が効くか**だけ見ていただけますか？ そこが OK なら、こちらは確実に組み上げられます」と促して待つ。
- **テンプレ自体が無い** → 「まず①②（検索の仕組み＋テンプレ）の用意が要ります。これは SF 画面での作業になります。用意できたら API 名を教えてください」と促して待つ。
  - （補助）どうしても AI 側でテンプレ作成まで踏み込む場合のみ、[../assets/patterns/retriever-prompt-template.genAiPromptTemplate-meta.xml](../assets/patterns/retriever-prompt-template.genAiPromptTemplate-meta.xml) の雛形＋有効化（[独自の落とし穴.md](独自の落とし穴.md) #11）に従う。ただし**検索が効くかの最終確認は人**に依頼する。

## Bot が確認のために裏でやってよいこと（任意）

ユーザーが API 名をくれたら、実在と入力名だけ確認してよい（推測しないため）：
```bash
sf project retrieve start --metadata "GenAiPromptTemplate:<API名>" --target-org <alias> --json
grep -nE "<apiName>|EinsteinSearch:|getEinsteinRetrieverResults|<status>|activeVersionIdentifier" \
  force-app/main/default/genAiPromptTemplates/<API名>.genAiPromptTemplate-meta.xml
```
- `activeVersionIdentifier` が無い／`status` が Published でも有効版が無いなら、①②が未完了の可能性 → 人に確認を促す（[独自の落とし穴.md](独自の落とし穴.md) #11）。
