# 委譲手順（底層スキル / sf CLI の駆動）

ユーザーには見せない**内部の実作業手順**。詳細な構文は `.agents/skills/developing-agentforce/` を参照。`sf` は `npx @salesforce/cli` でも可。全コマンドに `--json`。
このスキルは**②（エージェント作成）**を担う。①（テンプレ設定・動作確認）は人間が済ませている前提（[前提チェック.md](前提チェック.md)）。

## 0. 前提確認
```bash
sf config get target-org --json          # 対象組織が設定されているか
sf org display --target-org <alias> --json   # 接続 / API 66+ 確認
```

## フェーズ1：検証済みテンプレの受領と入力名の確認 ★②の起点

ユーザーから**①で動作確認済みのテンプレ API 名**を受け取る。実在と入力名を確認する（推測しないため）：
```bash
sf project retrieve start --metadata "GenAiPromptTemplate:<検証済みテンプレ名>" --target-org <alias> --json
grep -nE "<apiName>|<required>|EinsteinSearch:|getEinsteinRetrieverResults|<status>|activeVersionIdentifier" \
  force-app/main/default/genAiPromptTemplates/<検証済みテンプレ名>.genAiPromptTemplate-meta.xml
```
- 読み取る：入力 `<inputs><apiName>`（＝ `.agent` で `"Input:<apiName>"` と宣言）と各 `<required>`。
- `activeVersionIdentifier` が無い／有効版が無いなら①未完了の可能性 → 人間に差し戻し（ガードレール#11）。
- 標準オブジェクトを入力にするなら API 名も確認：`sf sobject describe --sobject <Object> --target-org <alias> --json`。

sf org list metadata --metadata-type GenAiPromptTemplate --target-org <alias> --json

## ＜補助＞フェーズ2b：レトリーバーの特定 ★CLIで list 不可（やむを得ず Bot がテンプレを作る場合のみ）

通常は不要（テンプレは①で人間が用意）。**どうしても Bot がテンプレを新設する**ときだけ使う。
**レトリーバーは `sf` で一覧できない。** それを**使っている既存テンプレ**から読み取って特定する（[実現可能性と制約.md](実現可能性と制約.md) C）。
```bash
# 1) その検索を使っていそうな既存テンプレを retrieve
sf project retrieve start --metadata "GenAiPromptTemplate:<既存テンプレ名>" --target-org <alias> --json
# 2) 中から レトリーバー参照と入力パラメータを読み取る
grep -nE "getEinsteinRetrieverResults|EinsteinSearch:|parameterName|searchText" \
  force-app/main/default/genAiPromptTemplates/<既存テンプレ名>.genAiPromptTemplate-meta.xml
```

**レトリーバーは `sf` で一覧できない。** 既存レトリーバーを使うには、それを**使っている既存テンプレ**から読み取って特定する（[実現可能性と制約.md](実現可能性と制約.md) C）。
```bash
# 1) その検索を使っていそうな既存テンプレを retrieve
sf project retrieve start --metadata "GenAiPromptTemplate:<既存テンプレ名>" --target-org <alias> --json
# 2) 中から レトリーバー参照と入力パラメータを読み取る
grep -nE "getEinsteinRetrieverResults|EinsteinSearch:|parameterName|searchText" \
  force-app/main/default/genAiPromptTemplates/<既存テンプレ名>.genAiPromptTemplate-meta.xml
```
読み取る値（**推測禁止・実値のみ使用**）：
- `invocable://getEinsteinRetrieverResults/<RetrieverRef>`（`templateDataProviders > definition`）
- `EinsteinSearch:<RetrieverRef>`（`referenceName`）と本文の `{!$EinsteinSearch:<RetrieverRef>.results}`
- 入力パラメータ名（`searchText` 等）と、それに渡している `{!$Input:<入力名>}`

## ＜補助＞フェーズ3b：レトリーバー入りプロンプト道具を新規作成する場合（原則①で人間が用意）

通常は不要。**やむを得ず Bot がテンプレを作る**ときだけ。作成後の**検索が効くかの最終確認は人間に依頼**する。テンプレ XML を直接作る。要点：
- `force-app/main/default/genAiPromptTemplates/<NewTemplate>.genAiPromptTemplate-meta.xml` を作成。
- 本文に **`{!$Input:<入力名>}`** と **`{!$EinsteinSearch:<RetrieverRef>.results}`** を必ず参照。
- `<templateDataProviders>` にフェーズ2bで特定した `invocable://getEinsteinRetrieverResults/<RetrieverRef>` と `searchText`（`{!$Input:<入力名>}`）を設定。
- `<status>Published</status>`。**`<versionIdentifier>` は書かない**（手書きは deploy 拒否）。
- deploy → **有効化**（ガードレール#11）：
```bash
sf project deploy start --metadata "GenAiPromptTemplate:<NewTemplate>" --target-org <alias> --json
# 付与された versionIdentifier を読む
sf project retrieve start --metadata "GenAiPromptTemplate:<NewTemplate>" --target-org <alias> --json
grep -n "versionIdentifier" force-app/main/default/genAiPromptTemplates/<NewTemplate>.genAiPromptTemplate-meta.xml
# トップレベルに <activeVersionIdentifier>= その値 を追記して再 deploy
sf project deploy start --metadata "GenAiPromptTemplate:<NewTemplate>" --target-org <alias> --json
```

## ＜参考＞道具種別ごとの組み込み（②Bob）

ユースケースに応じて必要な道具をそろえる。基盤（検索・認証・チャネル）は①人間。詳細は [道具カタログ.md](道具カタログ.md)。

- **プロンプト道具**（`generatePromptResponse://`）：①の検証済みテンプレを retrieve で入力名確認 → `.agent` に `"Input:xxx"` 宣言＋`with`＋`set`。
- **フロー道具**（`flow://`）：既存フローは API 名でバインド。新規なら `generating-flow` で Autolaunched Flow を生成→deploy→`flow://<Flow>`。入出力変数を確認。
- **Apex 道具**（`apex://`）：`generating-apex` で `@InvocableMethod`（予約名 `model/description/label` 回避・数値は `complex_data_type_name`）を生成→`sf project deploy start --metadata ApexClass:<Class>`→`apex://<Class>`。
- **外部API道具**（`externalService://`）：①で Named Credential 認証済み前提。`building-sf-integrations` で External Service 登録→`externalService://<Service>`。
- **検索（レトリーバー）**：①で用意済みを参照（`retriever://` か プロンプトテンプレ埋め込み）。新設・一覧は不可（補助フェーズ2b）。
- **担当の切替**：`@utils.transition to @subagent.<name>`（受付＝ハブに集約。別ルーターを作らない）。
- **人手へ引き継ぎ**：サービスエージェントのみ `@utils.escalate`＋`connection messaging:`（①でチャネル設定）。
- **構造化I/O**：複雑な入出力は `generating-custom-lightning-type`（CLT）でスキーマ生成→deploy。

## フェーズ4：構築

DX プロジェクトが無ければ作る（既にあれば省略）：
```bash
sf project generate --name sf-project --output-dir <dir>
```

1. **authoring-bundle 生成**（`<Api>` は snake_case の英数字、連続アンダースコア禁止）：
```bash
sf agent generate authoring-bundle --json --no-spec --name "<日本語ラベル>" --api-name <Api>
```
2. **`.agent` を記述**：生成された `aiAuthoringBundles/<Api>/<Api>.agent` を雛形から上書き。
   - 1担当・プロンプト道具中心 → [../assets/patterns/prompt-agent.agent](../assets/patterns/prompt-agent.agent)
   - 複数担当（ハブ&スポーク） → [../assets/patterns/multi-topic-agent.agent](../assets/patterns/multi-topic-agent.agent)
   - **[ガードレール.md](ガードレール.md) 必読**（#1 #2 #3 #5、複数道具・サービス時は #13〜#20 も）。
3. **下書きチェック**：
```bash
sf agent validate authoring-bundle --json --api-name <Api>
```
   落ちたら [ガードレール.md](ガードレール.md) の診断順序で修正→再実行。
4. **組織に配置**：
```bash
sf project deploy start --metadata AiAuthoringBundle:<Api> --json
# 新規にプロンプトテンプレ/フローも作った場合はそれらも deploy
```

## フェーズ5：テスト
```bash
# 試し会話を開始（session-id が返る。対話REPLではない）
sf agent preview start --json --use-live-actions --authoring-bundle <Api>
# 発話を送る
sf agent preview send --json --authoring-bundle <Api> --session-id <ID> -u "<日本語の発話>"
# 終了
sf agent preview end --json --session-id <ID>
```
- 結果（`result.messages[].message`）を**日本語でやさしく**ユーザーに伝える。トレース/JSON は見せない。
- **回答が空／「データが取得できない」のとき**は、道具が呼ばれて実行時に失敗している可能性。ガードレール「FunctionStep トレースの読み方」で `input` と `errors[].detail` を確認し、#11/#12 等を自己修正してから再テスト（ユーザーには「少し調整します」とだけ伝える）。
- **検索が0件続く**場合は①（テンプレ/レトリーバー）側の問題の可能性。正直に伝え、人間に①の見直しを促す（Bot は検索精度を保証しない）。

## フェーズ6：成果物の提示（②の出力）
- **作成した Agent の名前/概要**（developer_name・ラベル・担当・バインドしたテンプレ）と**動作テスト結果**をまとめる。
- **ブラッシュアップ案**を [../assets/ブラッシュアップ案-テンプレ.md](../assets/ブラッシュアップ案-テンプレ.md) の形で出す。観点：テンプレ本文／検索クエリ（`searchText`）／`resultCount`／担当の `instructions`／入力の渡し方／出力形式。③で人間が SF 画面で壁打ちしながら調整するための具体材料にする。

## ＜任意＞自動評価テスト（公開後の品質保証）
発話と期待動作を YAML で書き、繰り返し検証する。**有効化済みエージェントが対象**。観点・期待値の設計は人間（③）。
```bash
sf agent test create --json --spec specs/<Api>-testSpec.yaml --api-name <TestDef> --force-overwrite
sf agent test run --json --api-name <TestDef> --wait 5
sf agent test results --json --job-id <JOB_ID>   # 非同期時
```
- spec の書式・設計手法は `testing-agentforce` スキルに委譲。

## ＜任意＞本番監視（観測）
公開後の実セッションを観測し、誤振り分け・アクション失敗・低品質を分類して改善に回す。
- STDM（Agentforce Activity）有効化は**①人間（Data Cloud 画面）**。取得・分類・診断は `observing-agentforce` スキルに委譲。
