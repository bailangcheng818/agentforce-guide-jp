# 実現可能性と制約（役割分担と「作れる/作れない」）

このスキルは「自然言語の対話だけ」でエージェントを作る支援をする。だが**何でも作れるわけではない**し、**全工程を Bot がやるわけでもない**。
役割分担を理解し、できない部分・人間がやる部分は**先にユーザーへ正直に伝える**。後工程でつまずいてから気づくのを避けるのが目的。

> 具体名（テンプレ名・レトリーバー ID・組織名など）はプロジェクトごとに変わる。**ここでは固有名を例にしない**。実際の値は対象組織を調べて確定する。

## 役割分担（重要）

| 工程 | 担当 |
|---|---|
| ① レトリーバー設定＋**プロンプトテンプレートで動作確認**（検索が効くか・4論点の見極め） | **人間（SF画面）** |
| ② レトリーバーを使った**エージェントの作成・デプロイ・テスト・ブラッシュアップ案** | **このスキル（Bot）** |
| ③ 細かなチューニング（SF画面で壁打ちしながら調整） | **人間（Bob/LLM 補助）** |

- Bot は①を**代行しない**。検索の精度・データの良し悪しは**保証しない**（[前提チェック.md](前提チェック.md)）。
- 下の A〜E は、主に**①で人間が見極める観点**と、**②で Bot が守る境界**の両方を含む。

---

## A. ②でこのスキル（Bot）が作れるもの

- AIアシスタント本体、担当（サブ）、受付（ハブ）の自動構成。
- **①で人間が動作確認済みのプロンプトテンプレートを束ねる道具**：`generatePromptResponse://<検証済みテンプレ名>`。入力名 `"Input:xxx"` を正しく宣言してバインド。
- **自動処理の道具**：既存フローの再利用、または `generating-flow` スキルでの新規フロー。
- 標準オブジェクト（商談・取引先など）の項目を入力に組み込む（`sf sobject describe` で API 名を確認）。
- 下書きチェック（validate）→ 組織へ配置（deploy）→ 試し会話（preview）→ 公開（任意）→ **ブラッシュアップ案の提示**。

## B. ②で Bot が作れない／対象外（先に伝える）

- **レトリーバー（検索インデックス）の新設・動作保証は対象外**。①で人間が用意・確認したものを**参照するだけ**。
- **プロンプトテンプレートの新設は原則①（人間）**。検索が効くかの確認も人間。やむを得ず Bot が作る場合のみ C/委譲手順 フェーズ3b（有効化必須）。
- **新しい外部連携（コネクタ・認証・データソース接続）の構築は対象外**。例：Box・外部DB・Web サービス等を**新規に**つなぐ作業。
  - ただし、**すでに連携済み**の外部データ（外部ストレージを取り込んだ検索インデックス等）を、①のテンプレ経由で**参照**するのは可能。
- Apex・カスタムオブジェクト・カスタム項目の**ゼロ作成**、生産観測、Slack 等チャネル連携は v1 対象外。
- 公開（publish / activate）は**確認の上**でのみ実施。

## C. 検索（RAG）の制約 ＝ ①が人間である理由

**レトリーバーは CLI で一覧（list）も新設（このスキルでは）もできない。** だから「どの検索を使うか」を機械的に選ぶことはできず、**①で人間が SF 画面で用意・確認**するのが前提（[前提チェック.md](前提チェック.md)）。

Bot が②でやるのは「**①で確認済みのテンプレを正しくバインドする**」こと。そのために、ユーザー提供のテンプレ API 名で retrieve し、入力名（`<inputs><apiName>` ＝ `Input:xxx`）と、必要なら埋め込まれたレトリーバー参照（`invocable://getEinsteinRetrieverResults/<...>` / `{!$EinsteinSearch:<...>.results}`）を**読み取って確認**する（推測しない。詳細は [委譲手順.md](委譲手順.md)）。

→ **フェーズ1で確認すべきこと**：「検索を使うテンプレを①で**動作確認済み**か」「その**API 名**は何か」。未確認なら①に差し戻す。

## D. 「できないこと」をどう振る舞わせるか

エージェントには、**対応範囲外・データ不足の依頼**が来たときに、

- **作り話をさせない**（検索結果に無い内容を創作しない）。
- できない旨と、**人手で行う手順や代替**を**日本語で正直に案内**させる（off_topic / ambiguous 担当＋各担当の instructions に明記）。

→ **①で人間が見極める**：このテンプレが答えられない依頼の傾向。 **②で Bot が確認**：「できない依頼が来たら、どう振る舞ってほしいか（正直に断る／人に促す 等）」。

## E. UX・出力の要件

仕上がりの満足度は出力体験で決まる。**①で人間がテンプレ動作確認時に見極め**、**②で Bot が要件として軽く確認**する：

- **出力言語**：日本語（既定）。
- **出力形式**：箇条書き／表／見出し／必要なら HTML など。
- **長さ・粒度**：簡潔に要点か、詳しくか。
- **リンク・ID の扱い**：レコード URL を出すか、出すなら遷移先の正確さ（誤リンク防止）。
- **誰に見せるか**：社内担当向けの言い回しか、社外のお客様向けの丁寧さか。

| 確認項目 | 担当 | 注意・差し戻し条件 |
|---|---|---|
| 検索テンプレを**動作確認済み**か／その**API 名** | **①人間** → ②でBotが受領 | 未確認なら①へ差し戻し（[前提チェック.md](前提チェック.md)） |
| どこまで答えられるか／Box 等の外部データ | **①人間**（テンプレ実行で見極め） | 新規の外部連携は対象外（B） |

| 出力の形・言語・長さ | ①で見極め／②で確認 | （E） |
| 公開まで必要？ | ②Bot が確認 | 公開は確認の上（B） |
